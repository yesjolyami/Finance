# Формат JSON-бэкапа v5

Этот документ фиксирует формат, который текущий Python-сервер возвращает из
`GET /api/export` и принимает через `POST /api/import`. Денежные значения во всех
полях `*Cents` — целые числа в копейках. Даты имеют формат `YYYY-MM-DD`, время
создания — строка ISO 8601 в UTC.

## Верхний уровень

```json
{
  "version": 5,
  "exportedAt": "2026-01-02T03:04:05+00:00",
  "accounts": [],
  "categories": [],
  "transactions": [],
  "goals": [],
  "debts": [],
  "debtPayments": []
}
```

- `version` — номер формата экспорта, для этого документа всегда `5`;
- `exportedAt` — момент создания файла;
- остальные поля — массивы сущностей, описанных ниже.

## Сущности

### `accounts`

| Поле | Тип | Назначение |
| --- | --- | --- |
| `id` | string | Уникальный идентификатор |
| `name` | string | Название счета |
| `color` | string | HEX-цвет `#rrggbb` |
| `sortOrder` | integer | Порядок вывода |
| `system` | boolean | Признак встроенного счета |
| `kind` | string | `regular` или `savings` |
| `bank` | string | Название банка, может быть пустым |
| `owner` | string | Владелец, может быть пустым |
| `createdAt` | string | Время создания |

### `categories`

| Поле | Тип | Назначение |
| --- | --- | --- |
| `id` | string | Уникальный идентификатор |
| `type` | string | `expense` или `income` |
| `name` | string | Название категории |
| `color` | string | HEX-цвет `#rrggbb` |
| `budgetCents` | integer | Месячный лимит; для дохода всегда `0` |
| `system` | boolean | Признак встроенной категории |
| `createdAt` | string | Время создания |

### `transactions`

| Поле | Тип | Назначение |
| --- | --- | --- |
| `id` | string | Уникальный идентификатор |
| `type` | string | `expense`, `income` или `transfer` |
| `accountId` | string | Исходный счет |
| `toAccountId` | string \| null | Счет назначения только для перевода |
| `categoryId` | string \| null | Категория; `null` только для перевода |
| `amountCents` | integer | Положительная сумма в копейках |
| `date` | string | Дата операции |
| `note` | string | Комментарий, может быть пустым |
| `isBalanceAdjustment` | boolean | Операция ручной корректировки баланса |
| `createdAt` | string | Время создания |

### `goals`

| Поле | Тип | Назначение |
| --- | --- | --- |
| `id` | string | Уникальный идентификатор |
| `name` | string | Название цели |
| `targetCents` | integer | Целевая сумма |
| `savedCents` | integer | Уже накоплено, может быть `0` |
| `targetDate` | string | Целевая дата, может быть пустой |
| `color` | string | HEX-цвет `#rrggbb` |
| `archived` | boolean | Признак архива |
| `createdAt` | string | Время создания |

### `debts`

| Поле | Тип | Назначение |
| --- | --- | --- |
| `id` | string | Уникальный идентификатор |
| `person` | string | Имя человека |
| `direction` | string | `owe_me` или `i_owe` |
| `amountCents` | integer | Исходная сумма долга |
| `paidCents` | integer | Сумма связанных возвратов, вычисляется при экспорте |
| `leftCents` | integer | Остаток, вычисляется при экспорте |
| `dueDate` | string | Срок, может быть пустым |
| `note` | string | Комментарий, может быть пустым |
| `archived` | boolean | Признак архива |
| `createdAt` | string | Время создания |

`paidCents` и `leftCents` являются производными: при импорте итог повторно
вычисляется из `debtPayments`, а не доверяется этим двум полям.

### `debtPayments`

| Поле | Тип | Назначение |
| --- | --- | --- |
| `id` | string | Уникальный идентификатор |
| `debtId` | string | Ссылка на существующий долг |
| `amountCents` | integer | Положительная сумма возврата |
| `date` | string | Дата возврата |
| `note` | string | Комментарий, может быть пустым |
| `createdAt` | string | Время создания |

## Целостность и совместимость

- идентификаторы уникальны внутри каждого массива;
- ссылки на счета, категории и долги должны существовать в том же файле;
- файл обязан содержать хотя бы одну категорию доходов и одну категорию расходов;
- перевод ссылается на два разных счета и не имеет категории;
- обычная операция имеет категорию того же типа;
- `accounts` может отсутствовать только у старых бэкапов: тогда сервер создает
  стандартные счета;
- `goals`, `debts` и `debtPayments` могут отсутствовать у старых бэкапов и
  считаются пустыми массивами;
- импорт полностью заменяет данные в выбранной базе, поэтому проверять его нужно
  на отдельной временной БД или после создания резервной копии.

Текущий импорт сохраняет обратную совместимость и не отклоняет файл только по
значению `version`; фактическая совместимость определяется проверкой полей.
